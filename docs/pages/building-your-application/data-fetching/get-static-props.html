getStaticProps

페이지에서 getStaticProps라는 함수를 내보내면 Next.js는 빌드 시점에 getStaticProps가 반환한 프로퍼티를 사용하여 이 페이지를 미리 렌더링합니다.

import type { InferGetStaticPropsType, GetStaticProps } from 'next'

type Repo = {
  name: string
  stargazers_count: number
}

export const getStaticProps: GetStaticProps<{
  repo: Repo
}> = async () => {
  const res = await fetch('https://api.github.com/repos/vercel/next.js')
  const repo = await res.json()
  return { props: { repo } }
}

export default function Page({
  repo,
}: InferGetStaticPropsType<typeof getStaticProps>) {
  return repo.stargazers_count
}
export const getStaticProps = async () => {
  const res = await fetch('https://api.github.com/repos/vercel/next.js')
  const repo = await res.json()
  return { props: { repo } }
}

export default function Page({ repo }) {
  return repo.stargazers_count
}

렌더링 유형에 관계없이 모든 'prop'은 페이지 컴포넌트로 전달되며 클라이언트 측에서 초기 HTML에서 볼 수 있습니다. 이는 페이지가 올바르게 하이드레이션될 수 있도록 하기 위한 것입니다. 클라이언트에서 사용할 수 없어야 하는 민감한 정보를 props에 전달하지 않도록 주의하세요.

언제 getStaticProps를 사용해야할까?​

다음과 같다면 getStaticProps를 사용하는 것이 좋다:

  • 페이지 렌더링에 필요한 데이터는 사용자 요청에 앞서 빌드 시점에 사용할 수 있습니다.
  • 데이터는 헤드리스 CMS에서 가져옵니다.
  • 페이지가 사전 렌더링되어야 하며(SEO를 위해) 매우 빨라야 합니다 - getStaticProps는 HTML 및 JSON 파일을 생성하며, 이 두 파일은 성능을 위해 CDN에 캐시될 수 있습니다.
  • 데이터는 공개적으로 캐시될 수 있습니다(사용자별이 아님). 특정 상황에서는 미들웨어를 사용하여 경로를 다시 작성함으로써 이 조건을 우회할 수 있습니다.

getStaticProps는 언제 실행되나요?​

getStaticProps는 항상 서버에서 실행되며 클라이언트에서는 실행되지 않습니다. 이 도구를 사용하면 getStaticProps 내부에 작성된 코드가 클라이언트 측 번들에서 제거되었는지 확인할 수 있습니다(https://next-code-elimination.vercel.app/).

  • getStaticProps는 항상 next build 중에 실행됩니다.
  • getStaticProps는 fallback: true 사용 시 백그라운드에서 실행됩니다.
  • fallback: blocking 사용 시 초기 렌더링 전에 getStaticProps가 호출됩니다.
  • revalidate(재검증)을 사용할 때 백그라운드에서 getStaticProps가 실행됩니다.
  • getStaticProps는 revalidate() 사용 시 백그라운드에서 온디맨드 방식으로 실행됩니다.

Incremental Static Regeneration과 함께 사용하면 오래된 페이지가 재검증되는 동안 getStaticProps가 백그라운드에서 실행되고 새로운 페이지가 브라우저에 제공됩니다.

getStaticProps는 정적 HTML을 생성하므로 들어오는 요청(예: 쿼리 매개변수 또는 HTTP 헤더)에 액세스할 수 없습니다. 페이지에 대한 요청에 액세스해야 하는 경우 getStaticProps 외에 미들웨어를 사용하는 것을 고려하세요.

getStaticProps를 사용하여 CMS에서 데이터 가져오기​

다음 예는 CMS에서 블로그 게시물 목록을 가져오는 방법을 보여줍니다.

// 게시물은 빌드 시점에 getStaticProps()에 의해 채워집니다.
export default function Blog({ posts }) {
  return (
    <ul>
      {posts.map((post) => (
        <li>{post.title}</li>
      ))}
    </ul>
  )
}

// 이 함수는 서버 측에서 빌드 타임에 호출됩니다.
// 클라이언트 측에서는 호출되지 않으므로 직접 데이터베이스 쿼리를 수행할 수도 있습니다.
export async function getStaticProps() {
  // 외부 API 엔드포인트를 호출하여 게시물을 가져옵니다.
  // 모든 데이터 가져오기 라이브러리를 사용할 수 있습니다.
  const res = await fetch('https://.../posts')
  const posts = await res.json()

  // { props: { posts } }를 반환함으로써
  // Blog 컴포넌트는 빌드 시점에 'posts'를 프로퍼티로 받습니다.
  return {
    props: {
      posts,
    },
  }
}

getStaticProps API 참고 자료에는 getStaticProps와 함께 사용할 수 있는 모든 파라미터와 프로퍼티가 설명되어 있습니다.

server-side 코드 directly 작성​

getStaticProps는 서버 측에서만 실행되므로 클라이언트 측에서는 절대 실행되지 않습니다. 브라우저용 JS 번들에도 포함되지 않으므로 브라우저로 전송되지 않고 직접 데이터베이스 쿼리를 작성할 수 있습니다.

즉 (외부 소스에서 데이터를 가져오는) getStaticProps에서 API 경로를 가져오는 대신, 서버 측 코드를 getStaticProps에 바로(directly) 작성할 수 있습니다.

다음 예시를 살펴보겠습니다. API 경로는 CMS에서 일부 데이터를 가져오는 데 사용됩니다. 그런 다음 해당 API 경로를 getStaticProps에서 직접 호출합니다. 이렇게 하면 추가 호출이 발생하여 성능이 저하됩니다. 대신 lib/ 디렉터리를 사용하여 CMS에서 데이터를 가져오는 로직을 공유할 수 있습니다. 그런 다음 getStaticProps와 공유할 수 있습니다.

// 다음 함수는 `lib/` 디렉토리에서
// getStaticProps 및 API 경로와 공유됩니다.
export async function loadPosts() {
  // 외부 API 엔드포인트를 호출하여 게시물 가져오기
  const res = await fetch('https://.../posts/')
  const data = await res.json()

  return data
}
// pages/blog.js
import { loadPosts } from '../lib/load-posts'

// 이 함수는 오직 서버 사이드에서 실행됩니다
export async function getStaticProps() {
  // `/api` 경로를 가져오는 대신
  // `getStaticProps`에서 동일한 함수를 직접 호출할 수 있습니다.
  const posts = await loadPosts()

  // 반환된 Props는 페이지 컴포넌트로 전달됩니다.
  return { props: { posts } }
}

API 경로를 사용하여 데이터를 가져오지 않는 경우 fetch() API를 getStaticProps에서 직접 사용하여 데이터를 가져올 수 있습니다.

클라이언트 측 번들에서 Next.js가 제거한 내용을 확인하려면 next-code-elimination tool을 사용할 수 있습니다.

HTML과 JSON을 모두 정적으로 생성​

getStaticProps가 포함된 페이지가 빌드 시점에 미리 렌더링되면 페이지 HTML 파일 외에도 Next.js는 getStaticProps 실행 결과를 담은 JSON 파일을 생성합니다.

이 JSON 파일은 클라이언트 측 라우팅에서 next/link 또는 next/router를 통해 사용됩니다. getStaticProps를 사용하여 미리 렌더링 된 페이지로 이동하면 Next.js는 이 JSON 파일(빌드 시점에 미리 계산됨)을 가져와 페이지 컴포넌트의 프롭으로 사용합니다. 즉, 클라이언트 측 페이지 전환은 내보낸 JSON만 사용되므로 getStaticProps`를 호출하지** 않습니다.

Incremental Static Generation(증분 정적 생성)을 사용하는 경우 getStaticProps가 백그라운드에서 실행되어 클라이언트 측 탐색에 필요한 JSON을 생성합니다. 동일한 페이지에 대해 여러 요청이 수행되는 형태로 표시될 수 있지만 이는 의도된 것이며 최종 사용자 성능에는 영향을 미치지 않습니다.

getStaticProps는 어디에서 사용할 수 있나요?​

getStaticProps는 페이지에서만 내보낼 수 있습니다. 페이지가 아닌 파일인 _app, _document 또는 _error에서는 내보낼 수 없습니다.

이 제한의 이유 중 하나는 페이지가 렌더링되기 전에 React에 필요한 모든 데이터가 있어야 하기 때문입니다.

또한, getStaticProps 내보내기는 독립형 함수로 사용해야 하며, 페이지 컴포넌트의 프로퍼티로 getStaticProps를 추가하면 작동하지 않습니다.

알아두면 좋은 정보: custom app을 만든 경우, 링크된 문서에 표시된 대로 페이지 컴포넌트에 pageProps를 전달해야 하며, 그렇지 않으면 props가 비어 있게 됩니다.

Runs on every request in development​

개발 중(next dev)에는 모든 요청에 대해 getStaticProps가 호출됩니다.

미리보기 모드​

미리 보기 모드를 사용하면 일시적으로 정적 생성을 우회하고 빌드 시간 대신 요청 시간에 페이지를 렌더링할 수 있습니다. 예를 들어 헤드리스 CMS를 사용 중이고 초안을 게시하기 전에 미리 보고 싶을 수 있습니다.
