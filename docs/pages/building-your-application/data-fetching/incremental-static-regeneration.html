Incremental Static Regeneration

예시
  • Next.js Commerce
  • GitHub Reactions Demo
  • Static Tweet Demo

Next.js는 사이트를 빌드한 후에도 정적 페이지를 생성하거나 업데이트할 수 있도록 합니다. Incremental Static Regeneration(ISR)을 사용하면 전체 사이트를 다시 빌드하지 않고도 페이지별 정적 생성을 사용할 수 있습니다. ISR을 사용하면 정적 생성의 이점을 유지하면서 수백만 개의 페이지까지 확장할 수 있습니다.

참고: edge runtime은 현재 ISR과 호환되지 않지만, cache-control 헤더를 수동으로 설정하여 stale-while-revalidate를 활용할 수 있습니다.

ISR를 사용하려면 getStaticProps에 revalidate 속성을 추가하세요.

function Blog({ posts }) {
  return (
    <ul>
      {posts.map((post) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}

// 이 함수는 서버 측에서 빌드 시 호출됩니다.
// 새로운 요청이 들어오고 재검증(revalidation)이 활성화된 경우,
// 서버리스 함수에서 다시 호출될 수 있습니다.
export async function getStaticProps() {
  const res = await fetch('https://.../posts')
  const posts = await res.json()

  return {
    props: {
      posts,
    },
    // Next.js는 페이지를 다시 생성하려고 시도합니다.
    // - 요청이 들어온 경우
    // - 최대 10초마다 한 번씩
    revalidate: 10, // 초 단위 설정
  }
}

// 이 함수는 서버 측에서 빌드 시 호출됩니다.
// 만약 경로가 생성되지 않았다면,
// 서버리스 함수에서 다시 호출될 수 있습니다.
export async function getStaticPaths() {
  const res = await fetch('https://.../posts')
  const posts = await res.json()

  // posts를 기반으로 사전 렌더링하려는 경로를 가져옵니다.
  const paths = posts.map((post) => ({
    params: { id: post.id },
  }))

  // 이 경로들은 빌드 시간에만 사전 렌더링 됩니다.
  // { fallback: 'blocking' }는 페이지를 서버에서 렌더링합니다.
  // 경로가 존재하지 않는 경우에는 요청 시에만 제공됩니다.
  return { paths, fallback: 'blocking' }
}

export default Blog

빌드시 사전 렌더링 된 페이지로 요청이 발생하면, 캐시 된 페이지가 처음에 표시됩니다.

  • 초기 요청 이후 10초 전에 페이지에 대한 모든 요청도 캐시 되어 즉시 처리됩니다.
  • 윈도우가 열리고 10초가 지난 이후, 다음 요청은 여전히 캐시 된(오래된) 페이지를 보여줄 것입니다.
  • Next.js는 페이지를 백그라운드에서 재생성합니다.
  • 페이지가 성공적으로 생성되면, Next.js는 캐시를 무효화하고 업데이트된 페이지를 보여줍니다. 만약 백그라운드 재생성이 실패하면, 이전 페이지는 변경되지 않은 채로 남아있을 것입니다.

요청된 경로가 생성되지 않은 경우, Next.js는 첫 번째 요청에서 페이지를 서버 렌더링합니다. 이후 요청은 캐시에서 정적 파일을 제공합니다. Vercel의 ISR은 캐시를 전역적으로 유지하고 롤백을 처리합니다..

참고: 주요 데이터 공급 업체에서 기본적으로 캐싱이 활성화되어 있는지 확인하세요. 새로운 데이터를 가져와 ISR 캐시를 업데이트할 수 없게 될 수 있으므로 비활성화(예: useCdn: false 사용)해야 할 수도 있습니다. 캐싱은 요청되는 엔드포인트에서 Cache-Control 헤더를 반환할 때 CDN에서 발생할 수 있습니다.

요청에 따른 재검증​

60초의 재검증 시간을 설정하면 모든 방문자는 사이트의 동일한 생성 버전을 1분 동안 볼 수 있습니다. 캐시를 무효화하는 유일한 방법은 1분이 지난 후 해당 페이지를 방문한 사람입니다.

v12.2.0부터 Next.js는 수동으로 특정 페이지의 Next.js 캐시를 삭제하기 위한 On-Demand Incremental Static Regeneration을 지원합니다. 이에 따라 다음과 같은 경우에 사이트를 업데이트하기가 더 쉬워집니다.

  • Headless CMS에서 콘텐츠가 생성되거나 업데이트되는 경우
  • 전자상거래 메타데이터(가격, 설명, 카테고리, 리뷰 등) 변경되는 경우

getStaticProps 내부에서는 요청에 의한 재검증을 사용하기 위해 revalidate를 명시할 필요가 없습니다. revalidate가 생략된 경우, Next.js는 기본값인 false(재검증 없음)를 사용하고 revalidate()가 호출될 때만 페이지를 재검증합니다.

참고: 미들웨어는 On-Demand ISR 요청에 대해 실행되지 않습니다. 대신, 다시 유효화하려는 정확한 경로에서 revalidate()를 호출하세요. 예를 들어, pages/blog/[slug].js와 /post-1 -> /blog/post-1로 재작성하는 경우, res.revalidate('/blog/post-1')을 호출해야 합니다.

요청에 따른 재검증을 사용하는 방법​

먼저, Next.js 앱에서만 알고 있는 비밀 토큰을 생성하세요. 이 비밀 토큰은 재검증 API 라우트에 무단 액세스를 방지하는 데 사용됩니다. 다음 URL 구조를 사용하여 라우트에 액세스할 수 있습니다(수동으로 또는 웹훅을 사용).

https://<your-site.com>/api/revalidate?secret=<token>

다음으로, 애플리케이션에 비밀번호를 환경 변수로 추가하세요. 마지막으로, 재검증 API Route를 생성하세요.

export default async function handler(req, res) {
  // 요청이 유효한지 확인하기 위해 비밀번호를 확인하세요.
  if (req.query.secret !== process.env.MY_SECRET_TOKEN) {
    return res.status(401).json({ message: 'Invalid token' })
  }

  try {
    // 실제 경로여야 하며 재작성된 경로가 아니어야 합니다.
    // 예를 들어 "/blog/[slug]"의 경우 "/blog/post-1"이어야 합니다.
    await res.revalidate('/path-to-revalidate')
    return res.json({ revalidated: true })
  } catch (err) {
    // 오류가 있었다면,
    // Next.js는 마지막으로 성공적으로 생성된 페이지를 계속해서 보여줍니다.
    return res.status(500).send('Error revalidating')
  }
}

데모를 보고 요청에 따라 재검증을 하는 과정을 확인하고 피드백을 제공해 보세요.

개발 환경에서 요청에 따른 재검증 테스트하기​

next dev로 로컬에서 실행할 때마다 모든 요청에 대해 getStaticProps가 호출됩니다. on-demand ISR 구성이 올바른지 확인하려면 프로덕션 빌드를 생성하고 프로덕션 서버를 시작해야 합니다.

$ next build
$ next start

그런 다음 정적 페이지가 성공적으로 재검증되었음을 확인할 수 있습니다.

오류 처리 및 재검증​

getStaticProps 내부에서 백그라운드 재생성을 처리할 때 오류가 발생하거나 수동으로 오류를 발생시키는 경우, 마지막으로 성공적으로 생성된 페이지가 계속 표시됩니다. 다음 요청에서는 Next.js가 getStaticProps를 다시 호출하려고 합니다.

export async function getStaticProps() {
  // 만약 요청이 처리되지 않은 오류를 던진다면
  // Next.js는 현재 페이지를 무효화하지 않고
  // 다음 요청에서 getStaticProps를 다시 시도할 것입니다.
  const res = await fetch('https://.../posts')
  const posts = await res.json()

  if (!res.ok) {
    // 서버 오류가 발생하면 캐시가 다음 성공적인 요청까지
    // 업데이트되지 않도록 반환하는 대신 오류를 발생시키는 것이 좋습니다.
    throw new Error(`Failed to fetch posts, received status ${res.status}`)
  }

  // 요청이 성공하면 게시물을 반환하고,
  // 10초마다 다시 유효성을 검사합니다.
  return {
    props: {
      posts,
    },
    revalidate: 10,
  }
}

자체 호스팅 ISR​

Incremental Static Regeneration (ISR)은 next start를 사용할 때 자체 호스팅 Next.js 사이트에서 기본적으로 작동합니다.

이 접근 방식은 Kubernetes나 HashiCorp Nomad와 같은 컨테이너 오케스트레이터에 배포할 때 사용할 수 있습니다. 기본적으로 생성된 assets은 각 pod의 메모리에 저장됩니다. 이는 각 pod이 정적 파일의 자체 복사본을 가지고 있다는 것을 의미합니다. 특정 pod이 요청받을 때까지 오래된 데이터가 표시될 수 있습니다.

모든 pod에서 일관성을 보장하기 위해 인-메모리 캐싱을 비활성화할 수 있습니다. 이렇게 하면 Next.js 서버가 파일 시스템에서 ISR에 의해 생성된 assets만 활용하도록 지시됩니다.

Kubernetes pods(또는 유사한 설정)에서 공유 네트워크 마운트를 사용하여 다른 컨테이너 간에 동일한 파일 시스템 캐시를 재사용할 수 있습니다. 동일한 마운트를 공유함으로써 next/image 캐시를 포함하는 .next 폴더도 공유되어 재사용됩니다.

인-메모리 캐싱을 비활성화하려면, next.config.js 파일에서 isrMemoryCacheSize를 0으로 설정하세요.

module.exports = {
  experimental: {
    // Defaults to 50MB
    isrMemoryCacheSize: 0,
  },
}

참고: 공유 마운트가 구성된 방식에 따라 동시에 캐시를 업데이트하려는 여러 개의 pod 간의 경합 상태를 고려해야 할 수도 있습니다.
