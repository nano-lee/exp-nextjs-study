getStaticPaths

페이지에 동적 라우트가 있고 getStaticProps를 사용하는 경우, 정적으로 생성될 경로의 목록을 정의해야 합니다.

동적 라우트를 사용하는 페이지에서 getStaticPaths (정적 사이트 생성) 함수를 내보내면, Next.js는 getStaticPaths에서 지정한 모든 경로를 정적으로 사전 렌더링합니다.

import type {
  InferGetStaticPropsType,
  GetStaticProps,
  GetStaticPaths,
} from 'next'

type Repo = {
  name: string
  stargazers_count: number
}

export const getStaticPaths: GetStaticPaths = async () => {
  return {
    paths: [
      {
        params: {
          name: 'next.js',
        },
      }, // 아래 "paths" 섹션을 참조하세요.
    ],
    fallback: true, // false 또는 "blocking"
  }
}

export const getStaticProps: GetStaticProps<{
  repo: Repo
}> = async () => {
  const res = await fetch('https://api.github.com/repos/vercel/next.js')
  const repo = await res.json()
  return { props: { repo } }
}

export default function Page({
  repo,
}: InferGetStaticPropsType<typeof getStaticProps>) {
  return repo.stargazers_count
}
export const getStaticPaths = async () => {
  return {
    paths: [
      {
        params: {
          name: 'next.js',
        },
      }, // 아래 "paths" 섹션을 참조하세요.
    ],
    fallback: true, // false 또는 "blocking"
  }
}

export const getStaticProps = async () => {
  const res = await fetch('https://api.github.com/repos/vercel/next.js')
  const repo = await res.json()
  return { props: { repo } }
}

export default function Page({ repo }) {
  return repo.stargazers_count
}

getStaticPaths API 레퍼런스는 getStaticPaths와 함께 사용할 수 있는 모든 매개변수와 속성을 다룹니다.

언제 getStaticPaths를 사용해야 하나요?​

동적 라우트를 사용하여 페이지를 정적으로 사전 렌더링하는 경우 및 다음 조건에 해당하는 경우 getStaticPaths를 사용해야 합니다.

  • 데이터가 헤드리스 CMS에서 오는 경우
  • 데이터가 데이터베이스에서 오는 경우
  • 데이터가 파일 시스템에서 오는 경우
  • 데이터를 공개적으로 캐시할 수 있는 경우(사용자 특정이 아님)
  • 페이지가 사전 렌더링되어야 하며 매우 빠르게 동작해야 하는 경우 — getStaticProps는 HTML 및 JSON 파일을 생성하며, 이 둘 다 CDN에서 성능을 위해 캐시될 수 있습니다.

getStaticPaths는 언제 실행되나요?​

getStaticPaths는 프로덕션에서 빌드 중에만 실행되며, 런타임 중에는 호출되지 않습니다. 이 툴을 사용하여 getStaticPaths 내부에 작성된 코드가 클라이언트 측 번들에서 제거되었는지 확인할 수 있습니다.

getStaticProps는 getStaticPaths와 어떻게 함께 실행되나요?​

  • getStaticProps는 빌드 중에 getStaticPaths에서 반환된 paths에 대해 next build 동안 실행됩니다.
  • fallback: true를 사용할 때 getStaticProps는 백그라운드에서 실행됩니다.
  • fallback: blocking을 사용할 때 getStaticProps는 초기 렌더링 전에 호출됩니다.

어디에서 getStaticPaths를 사용할 수 있나요?​

  • getStaticPaths는 getStaticProps와 함께 반드시 사용되어야 합니다.
  • getStaticPaths는 getServerSideProps와 함께 사용할 수 없습니다.
  • getStaticProps도 사용하는 동적 라우트에서 getStaticPaths를 내보낼 수 있습니다.
  • 페이지가 아닌 파일(예: components 폴더)에서 getStaticPaths를 내보낼 수 없습니다.
  • 페이지 컴포넌트의 속성이 아닌 독립 함수로 getStaticPaths를 내보내야 합니다.

개발 중에 모든 요청에서 실행됩니다​

개발(next dev) 중에는, getStaticPaths가 모든 요청에 대해 호출됩니다.

요청 시 경로 생성​

getStaticPaths를 사용하면 빌드 중에 페이지를 생성하는 대신 fallback과 함께 요청 시 생성할 수 있습니다. 빌드 중에 더 많은 페이지를 생성하면 빌드 속도가 느려집니다.

paths에 대한 빈 배열을 반환함으로써 모든 페이지를 즉시 생성하는 것을 연기할 수 있습니다. 이는 특히 여러 환경에 Next.js 애플리케이션을 배포할 때 유용합니다. 예를 들어, 미리보기를 위해 모든 페이지를 즉시 생성하여 빌드 속도를 높일 수 있지만(프로덕션 빌드는 제외) 이는 수백/수천 개의 정적 페이지가 있는 사이트에 유용합니다.

export async function getStaticPaths() {
  // 이 값이 참일 때(미리보기 환경에서) 정적 페이지를 사전 렌더링하지 마세요.
  // (빌드 속도는 빠르지만, 초기 페이지 로드는 느림)
  if (process.env.SKIP_BUILD_STATIC_GENERATION) {
    return {
      paths: [],
      fallback: 'blocking',
    }
  }

  // 외부 API 엔드포인트를 호출하여 게시물을 가져옵니다.
  const res = await fetch('https://.../posts')
  const posts = await res.json()

  // 게시물을 기반으로 사전 렌더링하려는 경로를 가져옵니다.
  // 프로덕션 환경에서는 모든 페이지를 사전 렌더링합니다.
  // (빌드 속도는 느리지만, 초기 페이지 로드는 빠름)
  const paths = posts.map((post) => ({
    params: { id: post.id },
  }))

  // { fallback: false }는 다른 라우트는 404를 반환해야 함을 의미합니다.
  return { paths, fallback: false }
}

getStaticProps vs getStaticPaths​

getStaticProps와 getStaticPaths는 둘 다 Next.js에서 사용되는 데이터 가져오기 API이지만 각각 다른 목적과 사용 사례를 가지고 있으며 함께 사용되는 경우가 많습니다. 다음은 이 두 API를 비교한 것입니다 :

  1. 목적 :

    • getStaticProps: 페이지 컴포넌트에 필요한 데이터를 가져오고 해당 데이터를 사용하여 페이지를 정적으로 사전 렌더링하는 데 사용됩니다. 이 API는 데이터를 가져와 페이지의 초기 상태를 설정하는 데 주로 사용됩니다.
    • getStaticPaths: 동적 라우트를 사용하는 페이지에서 어떤 경로가 사전 렌더링되어야 하는지 정의하는 데 사용됩니다. 이 API는 정적 경로 목록을 생성하여 Next.js가 어떤 경로를 사전 렌더링할지 결정하는 데 사용됩니다.

  2. 사용 시점 :

    • getStaticProps: 빌드 시에 실행되며 빌드 중에 데이터를 가져오고 페이지를 사전 렌더링합니다. 클라이언트 측에서는 실행되지 않습니다.
    • getStaticPaths: 빌드 시에 실행되며 어떤 경로가 미리 렌더링될지를 정의합니다. 빌드 중에 실행되며 클라이언트 측에서 실행되지 않습니다. 개발 중에는 모든 요청에 대해 실행됩니다.

  3. 반환 값:

    • getStaticProps: 페이지 컴포넌트의 초기 상태를 설정하기 위해 반환된 데이터를 사용합니다. 반환 값은 { props: { ... } } 형식이어야 합니다.
    • getStaticPaths: 정적으로 렌더링할 경로 목록을 정의하기 위해 반환되는 값은 { paths, fallback } 형식입니다. paths에는 경로 목록이 포함되고, fallback에는 false, true, 또는 'blocking' 중 하나가 지정됩니다.

  4. 사용 사례:

    • getStaticProps는 페이지 렌더링에 필요한 데이터를 가져오는 데 사용되며, 데이터가 빌드 시점에 사용 가능하고 페이지의 초기 상태를 설정해야 하는 경우에 사용됩니다.
    • getStaticPaths는 동적 라우트를 사용하는 페이지에서 어떤 경로가 사전 렌더링되어야 하는지를 정의하며, 헤드리스 CMS 데이터, 데이터베이스 데이터 또는 파일 시스템에서 데이터를 가져와 경로를 동적으로 생성할 때 사용됩니다.

  5. 함께 사용:

    • getStaticPaths와 getStaticProps는 동적 라우트 페이지에서 함께 사용되는 경우가 많습니다. getStaticPaths로 경로 목록을 정의하고, getStaticProps에서는 해당 경로에 필요한 데이터를 가져와 페이지를 사전 렌더링합니다.

  6. 실행 시점 결정:

    • getStaticProps는 페이지 컴포넌트마다 실행되고, 페이지가 빌드되는 동안 한 번 실행됩니다.
    • getStaticPaths는 페이지마다 실행되지 않으며 페이지가 동적 라우트를 가지고 있는 경우에만 해당 페이지의 경로 목록을 생성합니다.

이 두 API는 함께 사용되어 페이지의 데이터 가져오기와 페이지의 정적 사전 렌더링을 가능하게 합니다. 페이지 컴포넌트가 동적 라우트를 가질 때 주로 getStaticPaths와 getStaticProps를 함께 사용합니다.
