보편적인 테스팅 툴인 Cypress, Playwright, Jest with React Testing Library를 사용하여 Next.js를 설정하는 방법을 알아봅시다.

Cypress​

Cypress는 End-to-End (E2E)와 Component Testing에 사용되는 테스트 툴입니다.

빠른 시작​

create-next-app을 with-cypress 예제로 사용하여 빠르게 시작할 수 있습니다.

npx create-next-app@latest --example with-cypress with-cypress-app

직접 설정하기​

Cypress를 시작하려면 먼저 cypress 패키지를 설치하세요

npm install --save-dev cypress

package.json에 Cypress 스크립트를 추가합니다.

{
    "scripts": {
        "dev": "next dev",
        "build": "next build",
        "start": "next start",
        "cypress:open": "cypress open"
    }
}

Cypress를 최초 실행하면 권장하는 폴더 구조를 사용하는 예제가 생성됩니다.

npm run cypress:open

생성된 예제와 Cypress 도큐멘테이션의 Writing Your First Test 섹션을 참고하면 Cypress에 익숙해지는 데 도움을 받을 수 있습니다.

E2E와 컴포넌트 테스트 중 어떤걸 사용해야 하나요?​

Cypress 문서에 두 테스트 타입의 차이점과 언제 어떤걸 사용해야 하는지에 대한 가이드가 있습니다.

첫 번째 Cypress E2E 테스트 만들어보기​

다음 두 개의 Next.js 페이지가 있다고 가정합시다.

import Link from "next/link";

export default function Home() {
    return (
        <nav>
            <h1>Homepage</h1>
            <Link href="/about">About</Link>
        </nav>
    );
}
export default function About() {
    return (
        <div>
            <h1>About Page</h1>
            <Link href="/">Homepage</Link>
        </div>
    );
}

navigation이 올바르게 작동하는지 확인하는 테스트를 추가해봅시다.

describe("Navigation", () => {
    it("should navigate to the about page", () => {
        // index page에서 시작합니다.
        cy.visit("http://localhost:3000/");

        // "about" 문자열을 포함하는 href 속성을 가진 링크를 찾아 클릭합니다.
        cy.get('a[href*="about"]').click();

        // 새로운 url은 "/about"을 포함해야 합니다.
        cy.url().should("include", "/about");

        // 새로운 페이지는 "About page"라는 h1을 포함해야 합니다.
        cy.get("h1").contains("About Page");
    });
});

cypress.config.js 설정 파일에 baseUrl: 'https://localhost:3000'을 추가해두면 cy.visit("https://localhost:3000/") 대신 cy.visit("/")을 사용할 수 있습니다.

첫 번째 Cypress 컴포넌트 테스트 만들어보기​

컴포넌트 테스트는 전체 앱을 번들링하거나 서버를 실행할 필요 없이 특정 컴포넌트를 빌드하여 마운트하는 방식으로 이뤄집니다. 이는 성능이 더 좋은 테스트를 가능하게 하면서 E2E 테스트와 동일한 시각적 피드백과 API를 제공합니다.

토막 상식: 컴포넌트 테스트는 Next.js 서버를 실행하지 않기 때문에 <Image />나 getServerSideProps같이 서버에 의존적인 기능은 기본적으로 동작하지 않습니다. 컴포넌트 테스트에서 이러한 기능들을 작동시키는 방법은 Cypress Next.js docs를 참고하세요.

이전 섹션과 동일한 컴포넌트라고 가정하고, 컴포넌트가 예상대로 렌더링하는지 검증하는 테스트를 추가해봅니다.

import AboutPage from "./about.js";

describe("<AboutPage />", () => {
    it("should render and display expected content", () => {
        // About 페이지 리액트 컴포넌트를 마운트합니다
        cy.mount(<AboutPage />);

        // 새로운 페이지는 "About page"라는 h1을 포함해야 합니다.
        cy.get("h1").contains("About Page");

        // 의도한 URL을 가진 링크가 존재하는지 검증합니다.
        // 링크 검증은 E2E 테스트에 더 적합합니다.
        cy.get('a[href="/"]').should("be.visible");
    });
});

Cypress 테스트 실행하기​

E2E 테스트​

Cypress E2E 테스트는 실제 Next.js 앱을 테스트하기 때문에 Cypress를 실행하기 전에 Next.js 서버가 먼저 실행되어야 합니다. 앱이 어떻게 동작할지 더 명확히 확인하기 위에 프로덕션 코드를 테스트하는 것을 권장합니다.

npm run build와 npm run start 스크립트를 통해 Next.js 앱을 실행한 후 다른 터미널에서 npm run cypress -- --e2e 스크립트를 실행하여 Cypress E2E 테스트를 실행합니다.

토막 상식: 또는 start-server-and-test 패키지를 설치하고 package.json에 "test": "start-server-and-test start http://localhost:3000 cypress" 스크립트를 추가하여 Next.js 프로덕션 서버와 Cypress를 한번에 실행할 수 있습니다. 새로운 변경사항이 있을 때마다 앱을 다시 빌드해야 하는 것을 잊지 마세요.

컴포넌트 테스트​

npm run cypress -- --component 스크립트를 실행하여 Cypress 컴포넌트 테스트를 실행합니다.

CI를 위한 준비사항​

지금까지의 방식으로는 Cypress가 대화형 브라우저를 실행시키기 떄문에 CI 환경에서는 실행하기 어렵다는 것을 깨달았을 것입니다. cypress run 명령어를 사용하면 Cypress를 헤드리스 모드로 실행할 수 있습니다.

{
    "scripts": {
        //...
        "e2e": "start-server-and-test dev http://localhost:3000 \"cypress open --e2e\"",
        "e2e:headless": "start-server-and-test dev http://localhost:3000 \"cypress run --e2e\"",
        "component": "cypress open --component",
        "component:headless": "cypress run --component"
    }
}

Cypress와 CI에 대해 더 알아보려면 다음 리소스를 참고하세요.

  • Cypress Continuous Integration Docs
  • Cypress GitHub Actions Guide
  • Official Cypress GitHub Action
  • Cypress Discord

Playwright​

Playwright는 Chromium, Firefox, WebKit을 하나의 API로 자동화할 수 있는 테스팅 프레임워크입니다. 모든 플랫폼에서 End-to-End (E2E)와 통합 테스트를 작성하는데 사용할 수 있습니다.

빠른 시작​

create-next-app을 with-playwright 예제로 사용하여 빠르게 시작할 수 있습니다. 이는 Playwright가 모두 설정된 Next.js 프로젝트를 생성합니다.

npx create-next-app@latest --example with-playwright with-playwright-app

직접 설정하기​

npm init playwright를 통해 기존 NPM 프로젝트에 Playwright를 추가할 수도 있습니다.

Playwright를 수동으로 시작하려면 @playwright/test 패키지를 설치하세요.

npm install --save-dev @playwright/test

package.json에 Playwright 스크립트를 추가합니다.

{
    "scripts": {
        "dev": "next dev",
        "build": "next build",
        "start": "next start",
        "test:e2e": "playwright test"
    }
}

첫 번쨰 Playwright E2E 테스트 만들어보기​

다음 두 개의 Next.js 페이지가 있다고 가정합시다.

import Link from "next/link";

export default function Home() {
    return (
        <nav>
            <Link href="/about">About</Link>
        </nav>
    );
}
export default function About() {
    return (
        <div>
            <h1>About Page</h1>
        </div>
    );
}

navigation이 올바르게 작동하는지 확인하는 테스트를 추가해봅시다.

import { test, expect } from "@playwright/test";

test("should navigate to the about page", async ({ page }) => {
    // index page에서 시작합니다.
    await page.goto("http://localhost:3000/");
    // "About" 문자열을 포함하는 요소를 찾아 클릭합니다.
    await page.click("text=About");
    // 새로운 url은 "/about"을 포함해야 합니다.
    await expect(page).toHaveURL("http://localhost:3000/about");
    // 새로운 페이지는 "About page"라는 h1을 포함해야 합니다.
    await expect(page.locator("h1")).toContainText("About Page");
});

playwright.config.js 설정 파일에 baseUrl: 'https://localhost:3000'을 추가해두면 page.goto("https://localhost:3000/") 대신 page.goto("/")를 사용할 수 있습니다.

Playwright 테스트 실행하기​

Playwrigt는 실제 Next.js 앱을 테스트하기 때문에 Playwright를 실행하기 전에 Next.js 서버가 먼저 실행되어야 합니다. 앱이 어떻게 동작할지 더 명확히 확인하기 위에 프로덕션 코드를 테스트하는 것을 권장합니다.

npm run build와 npm run start 스크립트를 통해 Next.js 앱을 실행한 후 다른 터미널에서 npm run test:e2e 스크립트를 실행하여 Playwright 테스트를 실행합니다.

토막 상식: 또는 webServer 기능을 사용하여 개발 서버가 시작되고 완전히 사용 가능할 때까지 Playwright가 기다리도록 할 수 있습니다.

CI를 위한 준비사항​

Playwright는 기본적으로 헤드리스 모드로 실행됩니다. 모든 Playwright 의존성을 설치하려면 npx playwright install-deps를 실행하세요.

Playwright와 CI에 대해 더 알아보려면 다음 리소스를 참고하세요.

  • Getting started with Playwright
  • Use a development server
  • Playwright on your CI provider
  • Playwright Discord

Jest와 리액트 테스팅 라이브러리​

Jest와 리액트 테스팅 라이브러리는 단위 테스팅에 종종 함께 사용됩니다. Next.js 앱에서 Jest를 사용하는 세 가지 방법이 있습니다.

  1. 우리가 제공하는 빠른 시작 예제 중 하나 사용하기
  2. Next.js 러스트 컴파일러와 함께 사용하기
  3. Babel과 함께 사용하기

다음 섹션에서 각각의 옵션으로 Jest를 설정하는 방법을 알아봅시다.

빠른 시작​

create-next-app을 with-jest 예제로 사용하여 빠르게 시작할 수 있습니다.

npx create-next-app@latest --example with-jest with-jest-app

Jest 설정하기 (러스트 컴파일러 사용)​

Next.js 12 릴리즈부터 Next.js에는 Jest를 위한 설정이 기본으로 제공됩니다.

Jest를 설정하려면 jest, jest-environment-jsdom, @testing-library/react, @testing-library/jest-dom을 설치하세요.

npm install --save-dev jest jest-environment-jsdom @testing-library/react @testing-library/jest-dom

프로젝트 루트 디렉터리에 jest.config.mjs 파일을 생성하고 다음 내용을 추가합니다.

import nextJest from "next/jest.js";

const createJestConfig = nextJest({
    // 테스트 환경에서 `next.config.js`와 `.env` 파일을 로드할 경로를 제공합니다.
    dir: "./",
});

// Jest에 전달할 커스텀 설정을 추가합니다.
/** @type {import('jest').Config} */
const config = {
    // 각 테스트를 실행하기 전에 추가할 더 많은 설정 옵션을 추가합니다.
    // setupFilesAfterEnv: ['<rootDir>/jest.setup.js'],

    testEnvironment: "jest-environment-jsdom",
};

// createJestConfig를 이런 식으로 내보내면 next/jest가 Next.js 설정을 비동기적으로 로드합니다.
export default createJestConfig(config);

다음 요소들은 next/jest가 내부적으로 자동으로 설정합니다.

  • SWC를 사용한 transform 설정
  • 스타일시트 (.css, .module.css, scss 변수), 이미지 임포트, next/font의 자동 모킹
  • .env (+ 모든 변수)를 process.env에 로드
  • 테스트에서 node_modules 무시
  • 테스트에서 .next 무시
  • SWC transform을 활성화하는 플래그를 위해 next.config.js 로드

Good to know: To test environment variables directly, load them manually in a separate setup script or in your jest.config.js file. For more information, please see Test Environment Variables.

토막 상식: 환경 변수를 직접 테스트하려면 별도의 설정 스크립트나 jest.config.js 파일에서 이를 수등으로 로드해야 합니다. 자세한 내용은 환경 변수 테스트를 참고하세요.

Jest 설정하기 (바벨 사용)​

러스트 컴파일러룰 사용하지 않는 경우 수동으로 Jest를 설정하고 babel-jest와 identity-obj-proxy 패키지를 추가로 설치해야 합니다.

다음은 Next.js에서 Jest를 구성하는 권장 옵션입니다.

module.exports = {
    collectCoverage: true,
    // node 14.x에서 v8 coverageProvider는 좋은 속도와 양호한 보고서를 제공합니다.
    coverageProvider: "v8",
    collectCoverageFrom: [
        "**/*.{js,jsx,ts,tsx}",
        "!**/*.d.ts",
        "!**/node_modules/**",
        "!<rootDir>/out/**",
        "!<rootDir>/.next/**",
        "!<rootDir>/*.config.js",
        "!<rootDir>/coverage/**",
    ],
    moduleNameMapper: {
        // Handle CSS imports (with CSS modules)
        // https://jestjs.io/docs/webpack#mocking-css-modules
        "^.+\\.module\\.(css|sass|scss)$": "identity-obj-proxy",

        // Handle CSS imports (without CSS modules)
        "^.+\\.(css|sass|scss)$": "<rootDir>/__mocks__/styleMock.js",

        // Handle image imports
        // https://jestjs.io/docs/webpack#handling-static-assets
        "^.+\\.(png|jpg|jpeg|gif|webp|avif|ico|bmp|svg)$/i": `<rootDir>/__mocks__/fileMock.js`,

        // Handle module aliases
        "^@/components/(.*)$": "<rootDir>/components/$1",
    },
    // 각 테스트를 실행하기 전에 추가할 더 많은 설정 옵션을 추가합니다.
    // setupFilesAfterEnv: ['<rootDir>/jest.setup.js'],
    testPathIgnorePatterns: ["<rootDir>/node_modules/", "<rootDir>/.next/"],
    testEnvironment: "jsdom",
    transform: {
        // babel-jest 트랜스파일을 next/babel 프리셋과 함께 사용하세요
        // https://jestjs.io/docs/configuration#transform-objectstring-pathtotransformer--pathtotransformer-object
        "^.+\\.(js|jsx|ts|tsx)$": ["babel-jest", { presets: ["next/babel"] }],
    },
    transformIgnorePatterns: [
        "/node_modules/",
        "^.+\\.module\\.(css|sass|scss)$",
    ],
};

각 구성 옵션에 대한 자세한 내용은 Jest 문서를 참고하세요.

스타일시트 및 이미지 임포트 핸들링

Stylesheets and images aren't used in the tests but importing them may cause errors, so they will need to be mocked. Create the mock files referenced in the configuration above - fileMock.js and styleMock.js - inside a __mocks__ directory:

스타일시트와 이미지는 테스트에 사용되지 않지만 임포트에서 에러가 발생할 수 있으므로 모킹이 필요합니다. 위에서 설정한대로 __mocks__ 디렉터리 안애 fileMock.js와 styleMock.js를 생성하세요.

module.exports = {
    src: "/img.jpg",
    height: 24,
    width: 24,
    blurDataURL: "data:image/png;base64,imagedata",
};
module.exports = {};

정적 애셋에 대한 자세한 내용은 Jest 문서를 참고하세요.

선택사항: 커스텀 매쳐로 Jest 확장하기

@testing-library/jest-dom에는 .toBeInTheDocument()와 같이 테스트 작성을 도와주는 편리한 커스텀 매쳐가 있습니다. Jest 설정 파일에 아래 옵션을 추가하면 모든 테스트에서 커스텀 매쳐를 사용할 수 있습니다.

setupFilesAfterEnv: ["<rootDir>/jest.setup.js"];

그런 다음 jest.setup.js에 다음 import를 추가하세요.

import "@testing-library/jest-dom";

extend-expect was removed in v6.0, so if you are using @testing-library/jest-dom before version 6, you will need to import @testing-library/jest-dom/extend-expect instead.

각 테스트 전에 더 많은 설정 옵션을 추가해야 하는 경우 위의 jest.setup.js 파일에 추가하는 것이 일반적입니다.

선택사항: 절대경로 임포트와 모듈 경로 별칭

프로젝트에서 모듈 경로 별칭을 사용하는 경우 jsconfig.json 파일에 있는 옵션을 jest.config.js 파일의 moduleNameMapper 옵션과 일치시켜서 Jest가 이를 해석할 수 있도록 해야합니다.

{
    "compilerOptions": {
        "module": "esnext",
        "moduleResolution": "node",
        "baseUrl": "./",
        "paths": {
            "@/components/*": ["components/*"]
        }
    }
}
moduleNameMapper: {
  '^@/components/(.*)$': '<rootDir>/components/$1',
}

직접 테스트 작성해보기​

package.json에 테스트 스크립트 추가하기

package.json에 Jest 실행 스크립트를 watch 모드로 추가합니다.

{
    "scripts": {
        "dev": "next dev",
        "build": "next build",
        "start": "next start",
        "test": "jest --watch"
    }
}

jest --watch 스크립트는 파일이 변경되면 테스트를 다시 실행합니다. 더 많은 Jest CLI 옵션은 Jest 문서를 참고하세요.

첫 번째 테스트 작성하기

이제 프로젝트에서 테스트를 실행할 준비가 되었습니다. Jest 컨벤션에 따라 프로젝트 루트 디렉터리에 __test__ 폴더를 추가하고 여기에 테스트 코드를 작성하세요.

아래는 <Home /> 컴포넌트가 heading을 성공적으로 렌더링하는지 확인하는 테스트를 추가하는 예시입니다.

import { render, screen } from "@testing-library/react";
import Home from "../pages/index";
import "@testing-library/jest-dom";

describe("Home", () => {
    it("renders a heading", () => {
        render(<Home />);

        const heading = screen.getByRole("heading", {
            name: /welcome to next\.js!/i,
        });

        expect(heading).toBeInTheDocument();
    });
});

선택사항으로, 스냅샷 테스트를 추가하여 <Home /> 컴포넌트에 예상치 못한 변경사항이 있는지 추적할 수 있습니다.

import { render } from "@testing-library/react";
import Home from "../pages/index";

it("renders homepage unchanged", () => {
    const { container } = render(<Home />);
    expect(container).toMatchSnapshot();
});

토막 상식: 페이지 라우터 내부의 모든 파일은 경로로 간주되므로 테스트 파일은 반드시 페이지 라우터 내부에 포함되어서는 안됩니다.

테스트 모음 실행하기

npm run test 스크립트를 통해 테스트 모음을 실행합니다. 테스트가 성공하거나 실패한 후에는 더 많은 테스트를 추가할 때 도움이 되는 대화형 Jest 명령어 목록이 표시됩니다.

더 자세한 내용은 아래 리소스를 참고하세요.

  • Jest Docs
  • React Testing Library Docs
  • Testing Playground - element match를 위한 좋은 테스팅 방법을 보여줍니다.
